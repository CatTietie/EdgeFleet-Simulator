# 时序统计异常检测与动态阈值需求规格

## 背景与目标
EdgeFleet-Simulator当前的告警规则仅支持静态阈值比较（如温度>80触发告警），无法适应设备正常运行基线的差异。夏季设备正常运行温度可能就接近80度，静态阈值导致大量误报；新设备上线没有历史数据时阈值无法合理设定。没有基于时序统计的异常检测，无法识别趋势性恶化（温度缓慢上升但未超阈值）。

需要实现基于时序统计的异常检测和动态阈值：从InfluxDB历史数据自动计算设备正常基线，支持动态阈值和趋势告警，减少误报并提前预警。

## 需求清单
- R-SA-01: 动态阈值规则：基于设备近N小时历史数据计算P95/P99作为阈值
- R-SA-02: 趋势告警：指标在滑动窗口内持续上升或下降超过斜率阈值时触发
- R-SA-03: 基线学习期：新设备上线前N个数据点不触发动态阈值告警，仅用静态阈值
- R-SA-04: DSL扩展：新增DynamicCompareExpr（比较对象为统计值而非字面量）
- R-SA-05: 基线缓存：统计结果缓存到Redis，TTL与计算周期对齐
- R-SA-06: 基线刷新：每小时从InfluxDB重新计算基线，增量更新Redis缓存
- R-SA-07: 前端规则编辑器支持动态阈值配置：选择统计函数、时间窗口、百分位数
- R-SA-08: 告警事件中记录触发时的实际阈值和统计上下文

## 现有架构约束
- 规则DSL支持CompareExpr（字段、运算符、比较值），需扩展为动态比较
- 求值器递归评估AST，需在求值时查询Redis获取动态阈值
- InfluxDB已有query_service支持时间范围查询和聚合
- 告警规则存储在PostgreSQL，DSL解析为AST

## 技术栈约束
- 后端：FastAPI + PostgreSQL + InfluxDB + Redis
- 规则引擎：自定义DSL + AST递归求值
- 时序数据：InfluxDB Flux查询
- 统计缓存：Redis Hash + TTL

## 数据模型变更
- AlarmRule的condition字段DSL扩展：dynamic_compare语法
- 新增 DeviceBaseline 模型：device_id, metric, p50, p95, p99, std_dev, sample_count, computed_at
- Redis Key：baseline:{org_id}:{device_id}:{metric} → JSON
- AlarmEvent扩展：trigger_context (JSONB) 存放实际阈值和统计信息
- 新增后台任务：每小时刷新所有在线设备的基线

## 流程设计
1. 基线计算：每小时后台任务 → 查询InfluxDB最近24小时数据 → 计算P50/P95/P99 → 写入Redis
2. 规则求值：遇到DynamicCompareExpr → 从Redis读取基线 → 替换为实际阈值 → 执行比较
3. 趋势检测：查询InfluxDB滑动窗口 → 线性回归计算斜率 → 超过阈值则触发
4. 学习期：设备上线不足N个数据点 → 跳过动态阈值规则 → 仅执行静态规则

## 边界条件与异常处理
- InfluxDB查询超时：使用Redis缓存的旧基线，标记为延迟更新
- 基线数据不足（<30个点）：回退到静态阈值
- 设备长时间离线后上线：基线过期需重新学习
- 趋势窗口内数据缺失：窗口向前扩展直到满足最小数据量

## 验收标准
- 配置动态阈值规则temperature > P95(24h)后，设备温度超过其自身24小时P95时触发告警
- 新设备上线前100个数据点不触发动态阈值告警，仅静态阈值生效
- 温度在2小时内持续上升超过斜率阈值时触发趋势告警
- 前端规则编辑器可选择动态阈值类型并配置统计参数
- InfluxDB不可用时使用缓存基线继续求值
